# 중복 이미지(균열) 탐지 아키텍처 연구 및 Colab 연동 방법

## 1. 개요
현재 Wall-E 프로젝트에서 YOLO 객체 탐지 시 고유 ID 값을 생성하여 중복 이미지를 방지하려는 로직이 드론 영상의 특성(흔들림, 각도 변화, 지연 등)으로 인해 한계를 보임.
이에 따라 **"데이터 증강을 통한 실시간 서브 모델 재학습"** 방식의 효용성을 분석하고, 현실적인 대안을 모색함.

---

## 2. 데이터 증강 및 실시간 재학습 방식 (안티 패턴 분석)

### 2.1 제안된 아이디어
- **아키텍처**: 1개의 메인 모델(YOLO) + 1개의 서브 필터링 모델 (총 2개)
- **로직**: 새로운 균열 발견 시 해당 이미지를 여러 장으로 데이터 증강(Augmentation)하여 서브 모델을 학습시키고, 이후 탐지되는 객체가 이 서브 모델을 통과하면 중복 이미지로 간주하여 저장하지 않음.

### 2.2 한계점 및 비추천 사유 (실시간 드론 스트리밍 환경 기준)
1. **파국적 망각 (Catastrophic Forgetting)**
   - 딥러닝 모델의 특성상 새로운 균열 데이터만으로 학습을 진행하면 **기존에 학습한 다른 균열들의 특징을 잊어버리는 치명적 한계** 발생.
   - 이를 해결하기 위해 과거 데이터까지 누적하여 학습(Replay Buffer)해야 하므로 데이터가 커지고 학습 시간이 무한정 늘어남.
2. **지연 시간 (Time Gap & Bottleneck)**
   - 5~10장 수준의 소규모 데이터로 1~2 에포크만 학습하더라도, 새로운 모델 가중치를 연산하고 업데이트하는 과정에서 수 초 이상의 딜레이 발생.
   - 초당 15~30프레임 이상을 처리해야 하는 실시간 메인 스트리밍 환경을 마비시킬 위험이 높음.
3. **One-Class 문제 (모드 붕괴)**
   - '정답(균열)'만 존재하고 '오답(배경, 다른 균열)' 데이터가 없는 상태에서 분류(Classifier) 모델을 학습시키면, 모든 입력에 대해 무조건 "정답"이라고만 출력하게 됨.
4. **시스템 오버헤드 (OOM 위험)**
   - 메인 추론 모델과 서브 학습 모델이 GPU 메모리(VRAM)를 동시에 점유할 경우 메모리 부족으로 서버 전체가 다운될 가능성이 높음.

## 3. 강력하고 현실적인 대안: 임베딩 기반 재식별 (Re-Identification)

"이전 객체와 현재 형태가 달라진 객체가 동일한가?"를 판단하는 가장 효율적인 방식은 모델의 '학습(Training)'이 아니라 **'특징 추출(Feature Extraction)'**입니다.

### 3.1 구현 방식 (임베딩 유사도 기반 필터링)
1. YOLO가 균열을 디텍팅하여 Bounding Box 영역을 크롭.
2. 크롭된 이미지를 아주 가벼운 특징 추출 전용 모델(Feature Extractor - 예: MobileNet, ResNet, CLIP Image Encoder 등)을 통과시켜 **1차원 벡터(임베딩 배열)** 1개로 변환.
3. 최근 DB 스토리지에 저장된 균열 이미지 임베딩 리스트와 방금 추출한 임베딩 간의 **코사인 유사도(Cosine Similarity)** 계산.
4. 유사도가 일정 임계값(예: 0.85 이상)을 넘기면, ID가 끊겼더라도 "동일한 균열"로 판단하고 이미지 저장을 드랍(Drop).

### 3.2 장점
- **극강의 실시간성**: 역전파 등 무거운 학습 과정이 없고 오직 벡터 내적 연산만 수행하므로 밀리초(ms) 단위로 완료됨.
- **안정성 유지**: 모델을 재로드할 필요 없이 기존 스트리밍 서버(FastAPI 등) 백그라운드에서 실시간으로 완벽하게 동작함.

## 4. 로컬 에디터(VS Code 등)와 Google Colab 환경 연동 방법

Colab을 파이썬 스크립트에서 일반 라이브러리(`import colab`)처럼 직접 호출하여 백그라운드 리소스로만 사용하는 것은 **구글의 정책상 막혀 불가능함.**
대신 원격 터널링을 이용해 "내 PC의 로컬 에디터에서 코딩하되, 코드가 실행되는 환경은 Colab의 가상 GPU로 연결"하는 편법을 주로 사용함.

### 4.1 가장 많이 사용하는 방식: VS Code Remote SSH 연결
VS Code를 Colab 서버의 GPU에 직접 터미널로 붙여서 로컬 파일처럼 작업하는 방식.

1. **Colab 웹 설정 (1회성 활성화)**
   - Colab 웹 브라우저 접속 후, GPU 할당 완료된 상태에서 `colab-ssh` 라이브러리를 통해 외부 접속용 SSH 터미널 주소 발급 (웹 브라우저는 백그라운드로 켜둠).
   ```python
   !pip install colab_ssh --upgrade
   from colab_ssh import launch_ssh_cloudflared
   launch_ssh_cloudflared("원하시는_비밀번호")
   ```
2. **VS Code 연결**
   - 로컬 VS Code에 `Remote - SSH` 확장 프로그램 설치.
   - Colab 셀이 응답한 SSH 호스트 및 비번 입력하여 원격 접속.
3. **환경 세팅 및 코딩**
   - 접속 후 VS Code의 터미널은 Colab 가상 머신의 리눅스 환경으로 변경됨.
   - 깃허브 코드를 받아오고, 코드 수정 후 `python scripts.py` 실행 시 연산은 Colab GPU에서 처리됨.

### 4.2 고려해야 할 치명적인 단점
- **수동 개입 필수**: 매번 Colab 접속, 세션 생성, 인증 비번 및 호스트 할당 작업을 사람이 수동으로 브라우저에서 진행해야 함. (코드 1줄 자동화 불가)
- **세션 날아감 (데이터 증발)**: 최대 12시간 지속 혹은 장시간 유휴 시 런타임 연결이 끊기며, 내부 데이터가 모두 초기화됨. (수시로 구글 드라이브나 외부 스토리지에 가중치 모델 `.pt` 자동 백업 로직을 필수로 넣어야 함)
