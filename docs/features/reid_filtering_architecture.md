# Wall-E Re-ID (균열 중복 필터링) 아키텍처 개선 문서

## 1. 개요
Wall-E 드론 앱은 실시간 비디오 스트리밍 중 YOLO 모델을 통해 외벽의 균열(Crack)을 탐지하고, 중복 저장을 방지하기 위해 **MobileNetV3 기반의 Re-ID (코사인 유사도)** 기능을 사용합니다. 

이 문서는 동일 프레임 내 다중 객체 동시 탐지 시 발생하던 False Positive(새로운 균열을 중복으로 오인하여 Drop하는 현상) 문제를 해결하기 위해 도입된 **Spatial/Temporal Filtering (공간/시간 필터링)** 아키텍처 업데이트 내역을 설명합니다.

---

## 2. 이전 로직의 한계점

### 2.1 기존 처리 흐름 (`v1`)
1. YOLO 모델이 1개의 이미지 프레임 안에서 N개의 박스를 찾음.
2. For 루프를 돌며 첫 번째 균열 이미지를 크롭(Crop) 후 임베딩(Vector) 추출.
3. 추출된 임베딩을 이전 프레임들의 캐시와 비교(Cosine Similarity). 
4. 새로운 균열이면 **즉시 캐시에 추가** (`reid_manager.add_to_cache()`).
5. 두 번째 균열 처리 시, **방금 캐시에 들어간 첫 번째 균열 임베딩과도 시각적 유사성을 비교**함.

### 2.2 발생하던 문제
- **오인 누락 현상 (False Positive Drop)**: 실제 물리적으로 멀리 떨어져 있는 완전히 별개의 균열이라 하더라도, 한 화면 안에 동시에 존재하고 시각적으로 비슷하게 생겼다면 첫 번째 균열만 저장되고 두 번째 균열은 중복 판정을 받아 DB 저장에서 누락됨.

---

## 3. 개선된 로직 (Spatial & Temporal Filters)

문제를 해결하기 위해 `reid_manager.py`의 캐시 메모리 구조를 확장하여, 단순 딥러닝 벡터 비교뿐만 아니라 메타데이터를 활용한 사전 필터링 단계를 추가했습니다.

### 3.1 메타데이터 확장
- **AS-IS:** `self.embedding_cache` = `(detection_id, embedding)`
- **TO-BE:** `self.embedding_cache` = `(detection_id, embedding, frame_id, track_id)`

이제 데이터가 언제(어느 프레임) 등장했고, YOLO ByteTrack 상에서 어떤 고유 번호(객체 추적)를 부여받았는지 함께 기억합니다.

### 3.2 2단계 사전 필터 적용

Re-ID 비교 과정(`is_duplicate`)에 진입할 때 무거운 행렬 연산을 수행하기 앞서 아래 두 가지 필터를 거칩니다.

#### 1) Temporal Filter (시간/추적 필터)
과거 캐시에 있던 객체 중 **현재 발견된 균열과 YOLO `track_id`가 정확히 동일한 캐시**가 발견되면, 100% 동일한 균열(드론이 계속 비추고 있는 같은 균열)이므로 **유사도 연산을 생략하고 즉시 중복(Duplicate) 처리**합니다.
*   **효과:** 무거운 임베딩 추출/비교 연산을 획기적으로 줄여 실시간 FPS를 최적화합니다.

#### 2) Spatial Filter (공간 필터)
캐시에 저장된 객체가 **현재 처리 중인 이미지와 동일한 `frame_id`(동일 시간의 캡처본)**를 가지고 있다면, **비교 루프를 건너뜁니다 (Skip).**
동일한 시간에 같은 화면상에 찍혔다는 것은 **물리적으로 다른 곳에 위치한 별개의 객체**임을 의미하기 때문에, 아무리 시각적으로 100% 흡사하더라도 절대 중복으로 취급하지 않습니다.
*   **효과:** 유사하게 생긴 주변의 다른 균열들이 연달아 누락되는 치명적인 데이터 소실 버그를 원천 차단합니다.

## 4. UI/UX 확장 기능 (Matched ID 표시)

로직 개선과 함께 사용자 경험 개선을 위해 프론트엔드 연동을 강화하였습니다.

- **백엔드 변경점 (`reid_manager.py` & `stream_manager_ai_only.py`)**: 
  중복 데이터로 판별 시(`max_sim >= 0.8`), 단순 `True` 값만 반환하지 않고 **가장 유사도가 높았던 원본 균열 사진의 고유 번호 (`matched_id`)** 를 찾아내어 WebSocket 페이로드로 전송합니다.
  
- **프론트엔드 변경점 (`live_streaming_screen_ai_mjpeg.dart`)**: 
  화면 우측 하단에 표시되는 실시간 오버레이 UI에 반영되어, "이 균열이 기존에 저장된 균열 중 어떤 것과 같은 녀석으로 취급되고 있는지"를 명시적으로 출력합니다. 
  *(예: `유사도: 92.5% (ID: 42)`)*

---

## 5. 최종 기대 효과 및 장점 요약

1. **데이터 신뢰성 보장 (No False Drops)**: 
   화면 내 완전히 비슷한 균열이 10개 동시에 표시되어도 단 한 건의 누락 없이 모두 고유 개체로 DB에 저장됨.
2. **AI 구조 연산 효율화**: 
   ByteTrack ID를 활용해 불필요한 Cosine Similarity 연산을 패스하므로 백엔드 CPU/GPU 리소스 낭비 감소.
3. **디버깅 편의성 및 신뢰도 향상**: 
   모바일 UI 상단에 자신이 어떤 번호(`ID`)의 균열로 취급받아 저장되지 않고 넘어가고 있는지 직관적으로 모니터링 가능.
