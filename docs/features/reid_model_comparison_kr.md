# Feature Extractor 모델 비교 분석: MobileNetV3-small vs ResNet18

Wall-E 프로젝트의 **실시간 균열 임베딩 기반 재식별(Re-ID)** 기능을 위해 고려되는 두 가지 경량 모델의 비교 분석입니다. 

---

## 1. 모델 개요

### 1.1 ResNet18 (Residual Network)
- **특징:** 딥러닝 비전 분야의 가장 기본적인 표준(Baseline) 모델 중 하나로, 잔차 연결(Residual Connection)을 사용하여 깊은 네트워크에서도 안정적인 성능을 보임.
- **파라미터 수:** 약 1100만 개 (11.7M)
- **임베딩(Feature) 벡터 크기:** 보통 512 차원
- **구조 강점:** 일반적인 이미지 분류(ImageNet 등)에서 검증된 탄탄한 특징 추출 능력을 보유.

### 1.2 MobileNetV3-small
- **특징:** 모바일 및 엣지 디바이스(Edge Device) 환경과 같이 컴퓨팅 파워가 극도로 제한된 환경에서 구동하기 위해 설계된 모델. Depthwise Separable Convolution을 활용해 연산량을 극한으로 압축.
- **파라미터 수:** 약 250만 개 (2.5M) - **ResNet18의 약 1/4 수준**
- **임베딩(Feature) 벡터 크기:** 보통 576 차원 (마지막 fc 제외 시)
- **구조 강점:** CPU 환경이나 저사양 하드웨어에서도 압도적인 빠른 속도로 동작.

---

## 2. 프로젝트 특성에 따른 상세 비교

Wall-E 시스템은 YOLOv11n을 사용해 이미 **고해상도 스트리밍(30fps 이상)**에서 메인 객체 탐지가 이루어지고 있습니다. Feature Extractor는 서브 로직으로 동작하므로 다음 세 가지 기준이 가장 중요합니다.

### ① 연산 속도 (Latency) & 실시간성
- **MobileNetV3-small (🏆 승리):**
  - 파라미터가 2.5M 수준으로 압도적으로 가볍기 때문에, 메인 스트림 프레임 레이트(FPS)에 전혀 영향을 주지 않습니다.
  - 별도의 GPU 리소스를 거의 차지하지 않으며, 심지어 **백엔드 서버의 CPU만으로도 실시간 피처 추출이 가능**한 수준입니다.
- **ResNet18:**
  - 11.7M의 파라미터를 가지고 있어 가벼운 축에 속하지만, 매번 잘라낸 균열 이미지를 통과시킬 경우 초당 수십 번의 연산이 추가될 때 메인 YOLOv11n의 스레드 성능에 미세한 병목(Bottleneck)을 유발할 수 있습니다.

### ② 임베딩 표현력 (Feature Representation)
- **ResNet18 (🏆 근소한 우위):**
  - 네트워크 파라미터가 많고 채널이 더 두껍기 때문에, 균열의 미세한 질감(Texture)이나 복잡한 배경 속에서의 패턴 차이를 구별하는 데 있어서 약간 더 높은 정밀도(유사도 분별력)를 가질 수 있습니다.
- **MobileNetV3-small:**
  - 모델을 압축하기 위해 연산을 간소화했기 때문에, 매우 복잡하고 정밀한 패턴 구별력은 ResNet 계열에 비해 미세하게 떨어질 수 있습니다. 
  - *그러나* "이미 찾은 같은 균열의 앵글이 살짝 틀어진 정보"를 매칭하는 수준의 **간단한 Re-ID 태스크에서는 전혀 부족함이 없는 충분한 표현력을 보여줍니다.**

### ③ 메모리 사용량 (Memory Footprint)
- **MobileNetV3-small (🏆 승리):**
  - 모델 크기가 약 10~15MB에 불과하고 VRAM 점유율이 50MB 이내로 매우 낮아, 메인 YOLO 모델 트레이닝/추론에 필요한 그래픽 카드 메모리를 방해하지 않습니다.
- **ResNet18:**
  - 모델 크기가 약 45MB로, VRAM 점유율이 MobileNet 대비 3~4배가량 높습니다.

---

## 3. 결론 및 추천 모델

**Wall-E 프로젝트에서는 [MobileNetV3-small]을 도입하는 것을 강력히 추천합니다.**

**[추천 사유]**
1. **서브 태스크의 목적성:** 재식별(Re-ID) 기능은 YOLO 모델의 성능이나 정확도를 깎아먹지 않는 **'투명하고 빠른 필터링'**으로 존재해야 합니다.
2. **충분한 성능:** 균열 이미지 자체는 개, 고양이, 자동차를 구별하는 복잡한 태스크가 아니라 단순한 선분/질감의 패턴 매칭이므로 MobileNetV3의 표현력(임베딩 퀄리티)만으로도 95% 이상의 정확한 필터링 코사인 유사도 연산이 가능합니다.
3. **확장성:** 만약 추후에 백엔드가 아닌 드론 자체 내장 보드(Raspberry Pi 등)나 아주 가벼운 노트북 서버 등 엣지단에 서버를 포팅하게 되더라도 소스 수정 없이 그대로 최적화된 속도로 구동할 수 있습니다.

### 📝 최종 파이프라인 적용 전략
- Pytorch `torchvision.models`에서 `mobilenet_v3_small(weights='DEFAULT')`를 다운로드하여 로드.
- 마지막 단계의 분류기(Classifier) 레이어를 떼어내고, 오직 `[1, 576]` 크기의 특성 벡터(Feature Vector)만 내뱉도록 커스텀.
- 이 가벼운 백본으로 초당 수 회~수십 회의 코사인 유사도를 백그라운드 스레드에서 연산 처리.
